## shogaishaのアーキテクチャ課題と解決策

現状の問題：Fat Controller
今のshogaishaは全処理がControllerに詰まっている。
[Request] → Controller（なんでもやる） → Model → DB
記事の言葉でいうとInterface層しか存在しない状態。

## 課題1：ルビキャッシュロジックの重複

index / list / show の3メソッドに同じコードが書かれている。

```
// ArticleController.php - 3箇所にコピーされている
$article->title_ruby = cache()->remember("article_title_ruby_{$article->id}", ...);
$article->excerpt_ruby = cache()->remember("article_excerpt_ruby_{$article->id}", ...);
```

「ルビ付きarticleを取得する」というUsecaseがControllerに3回散らばっている状態。
// ArticleController.php - 3箇所にコピーされてる

### 解決策：Serviceクラスに切り出す

```
// app/Services/ArticleService.php
class ArticleService {
    public function withRuby(Article $article): Article {
        $ruby = new RubyService();
        $article->title_ruby = cache()->remember(...);
        $article->excerpt_ruby = cache()->remember(...);
        return $article;
    }
}
```

Controllerは $this->articleService->withRuby($article) を呼ぶだけになる。


## 課題2：AuthControllerにEntry生成ロジックが混入

AuthController.php:59-69 — verify（ログイン検証）の中でEntryを作っている。

### 解決策 「認証」と「投稿」は全く別。AuthControllerがEntryの内部構造を知っているのは層の汚染。pending_entryの復元をEntryControllerに委ねる

```
// AuthController::verify
auth()->login($user);
$magicLink->delete();

// Entryの処理は委譲する
if (session()->has('pending_entry')) {
    return redirect()->route('entries.restore');
}
return redirect('/');

// EntryController::restore（新設）
public function restore() {
    $pending = session()->pull('pending_entry');
    // Entryの生成ロジックはここで完結
}
```

## 課題3：searchのビジネスロジックがControllerに直書き

検索クエリの組み立てがArticleControllerに書かれている。

```
Article::where('title', 'LIKE', "%{$q}%")
    ->orWhere('body', 'LIKE', "%{$q}%")
    ->latest()->get();
```

## 課題4 ：バリデーションがcontrollerの中に書かれている

store / update のバリデーションがControllerのメソッド内に直書き。記事のいうInterface層の責務（入力の検証）はFormRequestクラスに分離するのが正しい。

## 理想の構造

```
[Request]
    ↓
FormRequest（バリデーション = Interface層）
    ↓
Controller（ルーティングと返却のみ）
    ↓
Service（Usecase層：何を・どの順序で）
    ↓
Model / Repository（Domain + Infrastructure）
    ↓
DB / 外部API
```
